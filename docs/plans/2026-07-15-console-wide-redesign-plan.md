# Codex Deck 管理控制台全局统一改造方案

日期：2026-07-15

状态：首版视觉改造已实现，待真实 Tauri 业务链路验收

本方案替代仅针对“供应商配置”页的局部改造范围。此前供应商页面方案仍可作为实体卡片的细化参考，但后续实现以本总方案为准。

## 1. 已确认范围

用户已确认整个应用统一改造，包含：

- 全局侧边栏、顶部栏、页面标题和全局操作。
- 10 个主页面：概览、供应商配置、会话管理、工具与插件、Codex 增强、Zed 远程项目、脚本市场、安装维护、关于、设置。
- 供应商详情等二级页面。
- 确认弹窗、导入弹窗、编辑器、Toast、进度、加载、空状态和错误状态。
- 亮色和暗色主题。
- 应用允许的最小窗口宽度。

## 2. 明确依据、合理推断与非目标

### 2.1 代码和截图明确依据

- 前端当前有 10 个 `Route`，统一由 `App.tsx` 负责路由、数据状态和页面渲染。
- 全局框架由 `.shell`、`.sidebar`、`.topbar`、`.screen` 构成。
- 已有 `Button`、`Card`、`Badge`、`Input`、`Textarea` 和 Lucide 图标基础组件。
- 页面大量依赖 `Panel`、`CardHead`、`Toolbar`、`StatusRow`、`FeatureGroup`、`FeatureToggle` 等本地组件。
- 当前主页面普遍采用“大面板套小卡片”，信息密度、操作位置和状态表达不统一。
- `App.tsx` 约 292 KB，`styles.css` 约 67 KB；全量改造必须分阶段，不能在一次提交中同时重写视觉与业务逻辑。

### 2.2 本方案的推荐推断

- 采用 Cockpit Tools 的信息架构特征：集中工具栏、清晰状态卡片、分组设置行和稳定的全局导航。
- 不做像素级复制；Codex Deck 使用独立标识，保留现有业务字段和操作流程。
- 主交互色使用蓝色，表达选择和主要动作；绿色用于品牌细节点缀与成功状态。
- 侧边栏默认展开，窄窗口自动折叠为图标栏。用户不是技术人员，桌面宽屏下不应强制只显示图标。
- 四类页面模板覆盖所有路由：仪表盘、实体管理、设置、信息与诊断。

### 2.3 非目标

- 不改变 Rust 数据模型、Tauri 命令、配置生成、供应商切换和文件写入规则。
- 不新增 Cockpit Tools 的账号、配额、多实例等业务能力。
- 不修改 `Cargo.toml`、`package.json` 或引入新的 UI 框架。
- 不在视觉品牌改造中修改 `CodexPlusPlus` 内部程序标识、配置目录、crate 名称或数据路径。
- 不复制参考项目源码、图标资产或样式文件。
- 不在视觉改造中顺便调整长期业务功能。

## 3. 设计目标

1. 用户进入任意页面后，都能立即识别页面位置、当前状态和主要动作。
2. 同类动作在所有页面保持相同位置、图标、文案和反馈方式。
3. 信息密度适合桌面管理工具，减少大面积空白和无意义嵌套卡片。
4. 危险操作、异步进度、禁用原因和失败恢复路径必须清晰。
5. 亮暗主题、中文英文和长路径都不破坏布局。
6. 改造只影响展示层和交互组织，不改变现有业务结果。

## 4. 统一视觉系统

### 4.1 品牌标识

- 用户可见名称统一使用 `Codex Deck`，中文副标题使用“Codex 管理控制台”。
- 产品只服务 Codex，不使用“多 Agent 平台”等可能扩大产品边界的表达。
- 40px 主标识由抽象 `C` 形面板与终端提示符 `>` 组成，蓝色主体表达控制，绿色提示符表达连接正常。
- 不使用 OpenAI 官方图形；关于页和公开说明标注“第三方非官方 Codex 管理工具”。
- 亮暗主题共用同一标识结构，必须保证 40px 和应用图标尺寸下可辨识。
- 内部继续使用现有 `CodexPlusPlus` 技术标识，避免配置迁移和兼容性风险。

### 4.2 颜色

| 语义 | 亮色 | 暗色 | 用途 |
| --- | --- | --- | --- |
| 页面背景 | `#F3F5F9` | `#0F1218` | 应用工作区 |
| 导航背景 | `#FFFFFF` | `#141923` | 侧边栏、顶部栏 |
| 主表面 | `#FFFFFF` | `#181E28` | 工具栏、列表、卡片 |
| 次级表面 | `#F7F9FC` | `#202733` | 输入框、行悬停、分段控件 |
| 边框 | `#DCE3EC` | `#2D3645` | 控件和分隔线 |
| 主文字 | `#171B25` | `#F4F6FA` | 标题和正文 |
| 次级文字 | `#667085` | `#9BA7B8` | 说明、路径和时间 |
| 主操作 | `#2563EB` | `#5B8DEF` | 当前项、主按钮、焦点 |
| 品牌辅助/成功 | `#159F6B` | `#35C58A` | 标识终端符、健康、成功 |
| 警告 | `#D99012` | `#F0B34A` | 高延迟、需注意 |
| 危险 | `#D64545` | `#F06B6B` | 删除、错误 |

规则：颜色必须表达语义，不能仅用于装饰；当前选择用蓝色，成功状态用绿色，二者不能混用。

### 4.3 字体与间距

- 字体：沿用 Inter、Segoe UI、Microsoft YaHei。
- 页面标题：22px / 700。
- 页面说明：14px / 400。
- 区域标题：16px / 700。
- 卡片或行标题：14px / 650。
- 正文：14px；辅助文字：12px。
- 间距基线：4、8、12、16、20、24、32px。
- 不使用随视口缩放的字体，不使用负字距。

### 4.4 圆角、阴影和动效

- 卡片和工具栏最大圆角 8px。
- 输入框、按钮和菜单圆角 6px 或 8px。
- 状态标签和开关轨道可使用胶囊形状。
- 阴影只用于浮层、工具栏和当前实体，普通页面分区优先用边框与留白。
- 动效：120ms 反馈、180ms 常规切换、240ms 页面进入。
- 尊重 `prefers-reduced-motion`。

### 5.1 侧边栏

- 宽屏默认 220px，显示分组、图标和文字。
- 小于 960px 自动折叠为 68px 图标栏，使用 Tooltip 显示名称。
- 当前路由使用蓝色左侧指示、浅蓝背景和蓝色图标。
- 更新提示放在品牌区，系统状态放在侧边栏底部，不挤占页面内容。
- 保留现有 10 个入口，但按“工作台 / 扩展 / 系统”分组。

### 5.2 顶部栏

- 高度 76px，左侧固定页面标题和一句说明。
- 右侧保留语言、主题、重启 Codex Deck、刷新当前页面。
- 窄窗口只保留图标；重启操作使用图标和 Tooltip。
- 顶部栏可粘滞，但不能覆盖页面滚动内容。

### 5.3 页面主体

- 默认内边距 24px，最小宽度下降到 16px。
- 不给整个页面套装饰卡片；页面分区使用标题、分隔线和连续表面。
- 卡片只用于可重复实体、指标和真正独立的工具。
- 页面主操作始终位于工具栏右侧。

## 6. 共用组件

### 6.1 基础组件

- `AppShell`：侧边栏、顶部栏、工作区和响应式状态。
- `PageHeader`：标题、说明、全局动作。
- `PageToolbar`：搜索、视图、筛选、排序、主操作。
- `SegmentedControl`：互斥页面模式或标签页。
- `StatusBadge`：正常、警告、错误、禁用、进行中。
- `Switch`：所有二元设置统一为开关，不再使用方形复选框表达设置。
- `MetricTile`：只用于少量关键数字和状态。
- `DataTable`：密集实体列表、批量选择、列操作。
- `EntityCard`：供应商、市场脚本等需要比较的重复实体。
- `SettingsSection` / `SettingRow`：连续设置列表和行级控件。
- `InlineAlert`：环境冲突、风险和失败恢复。
- `ProgressRow`：异步任务进度、结果和重试。
- `EmptyState` / `Skeleton`：空数据和加载状态。
- `Modal` / `Drawer`：确认、小型编辑和复杂详情。
- `Toast`：短反馈，不承载必须处理的错误。

### 6.2 行为规则

- 每个页面最多一个主要按钮。
- 编辑、复制、测试、刷新、删除等熟悉动作优先使用 Lucide 图标和 Tooltip。
- 删除使用红色，只在确认弹窗中使用明确的对象名称和影响范围。
- 加载期间禁用重复提交；完成后在原位置显示结果，同时使用 Toast 简短反馈。
- 长路径和 URL 使用中等宽字体、单行省略与可复制操作。
- 搜索、筛选和临时排序不能意外修改持久数据。

## 7. 四类页面模板

### 7.1 仪表盘型

适用：概览。

- 顶部 4 个关键指标：Codex 版本、应用状态、入口状态、Watcher 状态。
- 中部为问题优先的健康列表；全部正常时只展示简洁成功状态。
- 右侧或下方展示最近启动信息和关键端口。
- 主操作“启动 Codex”；次操作“检查健康”“修复入口”。

### 7.2 实体管理型

适用：供应商、会话、工具与插件、Zed 项目、脚本市场。

- 顶部统一工具栏：搜索、视图、筛选、排序、刷新、添加。
- 数据较少且需比较时使用卡片；数据较多或需批量操作时使用表格。
- 详情使用独立页面或右侧抽屉，不在列表卡片内展开复杂表单。

### 7.3 设置型

适用：Codex 增强、设置、安装维护。

- 使用分段导航或左侧锚点切换设置组。
- 每组使用连续设置行和分隔线，避免卡片套卡片。
- 保存动作固定在页面底部操作栏；无修改时禁用。
- 依赖关系、禁用原因和“需重启生效”使用行内状态说明。

### 7.4 信息与诊断型

适用：关于、更新、日志、诊断。

- “关于 / 更新 / 日志 / 诊断”使用页面内标签页。
- 版本和更新状态优先，外部链接为次要操作。
- 日志与诊断使用固定高度查看器，提供刷新、复制、打开位置。

## 8. 逐页面蓝图

### 8.1 概览

- 指标条：版本、Codex App、快捷入口、Watcher。
- 健康列表只突出问题；路径放在展开详情或复制操作中。
- 最近启动显示状态、消息、Debug/Helper 端口和时间。
- 保留检查、修复、启动和打开关于。

### 8.2 供应商配置

- 延续已确认的搜索、卡片/列表、类型筛选、手动排序和总开关。
- 普通供应商卡片显示协议、Base URL、模型数和延迟。
- 聚合供应商显示策略、成员数和健康成员数。
- 当前供应商蓝色强边框；成功健康使用绿色。
- 详情页使用返回按钮、标题、状态和粘滞保存操作栏。

### 8.3 会话管理

- 顶部指标：总数、未归档、已归档、同步目标。
- 工具栏：搜索、归档筛选、更新时间排序、刷新、修复历史会话。
- 会话使用表格，列出标题、项目、Provider、更新时间、状态和操作。
- 批量选择后显示上下文操作栏；删除必须明确备份和文件影响。
- 自动修复设置移入“修复设置”抽屉，不占据列表上方的大面积空间。

### 8.4 工具与插件

- MCP、Skills、Plugins 使用分段标签与数量。
- 当前标签下提供搜索、状态筛选、添加和刷新。
- 项目使用紧凑列表：名称、来源、状态、启停、编辑、删除。
- 编辑器使用抽屉或模态框，保留 TOML 内容和现有保存逻辑。

### 8.5 Codex 增强

- 顶部显示增强总开关、运行模式和 Computer Use Guard。
- 页面内标签：插件与模型、对话与输入、界面与启动、远程项目。
- 每项使用设置行：标题、简短说明、状态标签、开关或动作。
- “需重启”“模式不适用”“系统强制启用”等状态必须显式显示。
- 插件市场修复和远程缓存进度跟随对应设置行，不放到页面底部。

### 8.6 Zed 远程项目

- 顶部显示发现项目数、最近项目数、默认打开策略。
- 项目按“已发现 / 最近使用”分段或筛选。
- 表格列出名称、主机、路径、来源和最近使用时间。
- 打开方式收进菜单；主动作使用默认策略，复制 URL 和移除为图标操作。

### 8.7 脚本市场

- 标签分为“市场 / 本地”。
- 市场工具栏提供搜索、分类、已安装筛选、刷新和打开仓库。
- 市场脚本使用卡片展示名称、版本、来源、说明和安装状态。
- 本地脚本使用表格，提供启停、来源、路径、更新和删除。

### 8.8 安装维护

- 顶部健康摘要统一展示 Codex App、入口、Watcher 和路径状态。
- 页面内标签：检查与修复、入口、Watcher、应用路径、手动启动。
- 危险的卸载、清除路径与移除 Watcher 使用次级危险按钮和确认弹窗。
- 检查和修复结果在当前区域显示，不依赖 Toast 承载长信息。

### 8.9 关于

- 页面内标签：关于、更新、日志、诊断。
- 关于页显示名称、版本、仓库和社区链接。
- 更新页显示当前/最新版本、发布日期、更新状态和安装进度。
- 日志和诊断使用统一查看器，支持复制和刷新。

### 8.10 设置

- 页面内标签：常规、模型与图片、Stepwise、启动参数。
- 设置使用连续行：标题和说明在左，控件在右。
- 文件路径使用“路径 + 打开/选择”组合控件。
- 页面底部粘滞保存栏显示“未修改 / 有未保存修改 / 保存中 / 已保存”。
- 主题与语言保留在设置页，同时继续提供顶部快捷入口。

## 9. 详情、弹窗与系统状态

### 9.1 详情页

- 使用页面级返回按钮和面包屑，不在原卡片内无限展开。
- 标题区展示对象名称、类型、当前状态和主要动作。
- 表单分区连续排列，保存栏粘滞在底部或顶部。

### 9.2 弹窗

- 确认弹窗：最大宽度 460px。
- 一般表单：最大宽度 640px。
- 复杂编辑器：最大宽度 860px，必要时使用右侧抽屉。
- 关闭按钮使用 `X` 图标并提供可访问名称。
- 删除确认按钮必须显示对象名称或数量。

### 9.3 页面状态

- 首次加载：保留稳定布局的 Skeleton。
- 空状态：说明当前没有数据，并提供唯一下一步动作。
- 部分失败：保留成功内容，在失败区域显示重试。
- 全页失败：显示错误摘要、可复制详情和重试。
- 禁用：必须说明原因，不能只降低透明度。

## 10. 响应式与无障碍

- `>= 1280px`：展开侧边栏，实体卡片最多三列。
- `960–1279px`：展开或紧凑侧边栏，工具栏允许两行，卡片两列。
- `760–959px`：图标侧边栏，工具栏分组换行，卡片单列。
- 不支持小于当前应用 760px 的宽度，避免引入伪移动端布局。
- 所有图标按钮有 `title` 和 `aria-label`。
- 所有表单有可关联 Label；开关可通过键盘操作。
- 焦点样式使用蓝色外圈，不能仅依靠颜色区分状态。
- 中英文、长 URL、Windows/macOS 路径均纳入验收样本。

## 11. 前端结构建议

不先重写业务状态。先把展示层从 `App.tsx` 中逐步拆出：

```text
src/
├─ components/
│  ├─ layout/          AppShell、Sidebar、PageHeader
│  ├─ ui/              Button、Badge、Switch、Tabs、Table、Modal
│  └─ shared/          PageToolbar、SettingRow、StatusSummary
├─ pages/
│  ├─ OverviewPage.tsx
│  ├─ ProvidersPage.tsx
│  ├─ SessionsPage.tsx
│  └─ ...
├─ styles/
│  ├─ tokens.css
│  ├─ shell.css
│  ├─ components.css
│  └─ pages/
└─ App.tsx             保留状态、命令调用和路由装配
```

原则：

- 现有 `Actions` 和 Tauri 调用留在 `App.tsx`，页面通过 Props 使用。
- 先抽纯展示组件，再迁移页面；不在同一提交里改业务行为。
- 新样式统一挂在 `.console-v2` 根节点，迁移期间避免污染旧页面。
- 全部页面迁移完成后再删除旧样式；删除前单独确认并核对引用。

## 12. 分阶段实施计划

### 阶段 0：行为基线

- 记录 10 个页面、详情页和弹窗的当前截图。
- 建立功能矩阵：入口、动作、状态、后端命令、数据文件影响。
- 明确每个页面的真实链路验收项。

### 阶段 1：设计系统和全局框架

- 新增颜色、排版、间距、动效 Tokens。

## 14. 主要风险与控制

| 风险 | 控制方式 |
| --- | --- |
| `App.tsx` 过大导致改动互相影响 | 先抽展示组件，业务状态与命令调用暂不移动 |
| 全局 CSS 选择器污染未迁移页面 | 使用 `.console-v2` 根作用域和按页样式文件 |
| 卡片化导致信息密度下降 | 仅实体和指标用卡片，密集数据使用表格和连续设置行 |
| 排序/筛选改变持久顺序 | 明确临时视图状态与持久数据边界 |
| 暗色主题后补导致返工 | 每个组件在阶段 1 同时完成双主题 |
| 视觉通过但真实功能回归 | 每页绑定真实命令、文件、数据库和日志验收项 |
| 参考项目许可证不明确 | 只借鉴布局原则，不复制源码与资产 |
| 使用 Codex 名称造成官方产品误解 | 使用独立标识，并在关于页和公开说明中标注第三方非官方工具 |

## 15. 已确认的开发条件

进入产品代码修改前，需要用户确认以下推荐方向：

1. 使用“蓝色主操作 + 绿色品牌/成功”的双色语义。
2. 侧边栏宽屏展开、窄窗口自动折叠。
3. 采用四类页面模板，而不是把所有页面都做成卡片网格。
4. 按六个阶段逐步交付和验收，不做一次性大提交。
5. 用户可见品牌使用 `Codex Deck`，内部技术标识暂不迁移。

用户已明确确认按总设计方案先完成一版；当前首版保留既有业务协议，后续按阶段继续细化组件与真实链路验收。

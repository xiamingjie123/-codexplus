# Codex Deck 供应商配置页 Cockpit 风格改造计划

日期：2026-07-15

状态：已并入全局总方案，仅保留供应商页面细化参考

## 1. 业务目标

把 Codex Deck 的“供应商配置”页改造成更适合多供应商管理的工作台：用户能快速搜索、比较、检测、切换和编辑供应商，同时保留现有配置写入、聚合供应商、第三方导入与拖动排序能力。

视觉结构参考 `jlcodes99/cockpit-tools` 的 Codex 账号页，但字段和操作必须服从 Codex Deck 的供应商业务，不能照搬账号配额与多实例功能。

## 2. 已有明确依据

### 当前 CodexPlusPlus 页面（内部技术标识）

- `RelayScreen` 已提供总开关、添加普通供应商、添加聚合供应商、第三方导入和供应商列表。
- `RelayProfileList` 已提供拖动排序。
- `SortableRelayProfileCard` 已提供延迟检测、设为当前、请求测试、编辑和删除。
- 当前供应商数据可展示名称、供应商类型、协议、Base URL、模型配置摘要、聚合成员和延迟状态。
- 供应商详情页已有独立编辑流程，本轮无需重写配置表单。

代码依据：

- `apps/codex-plus-manager/src/App.tsx`：`RelayScreen`、`RelayProfileList`、`SortableRelayProfileCard`
- `apps/codex-plus-manager/src/styles.css`：`.relay-profile-*`、`.relay-add-row`、`.relay-latency`

### 参考项目

- 参考页使用“搜索 + 视图切换 + 筛选 + 排序 + 快捷动作”的集中工具栏。
- Codex 账号以卡片网格展示，当前项使用强边框，状态、核心指标和行级操作在一张卡片内完成。
- 官方参考截图：`cockpit-tools/docs/images/codex_list.png`。

注意：参考仓库的公开元数据和根目录未识别到明确许可证文件。本方案只借鉴信息架构和交互模式，不复制其源码、图标资产或样式代码。

## 3. 推荐方案

### 范围

- 只重做“供应商配置”内容区。
- 全局侧栏、顶栏、主题和品牌遵守 `2026-07-15-console-wide-redesign-plan.md`，本页不单独定义另一套框架。
- 第一阶段保留现有供应商详情页，不同时重做编辑表单。
- 不引入账号管理、账号配额、多实例或参考项目的其他业务能力。

### 页面结构

1. 顶部工具栏
   - 按名称或 Base URL 搜索。
   - 卡片/列表视图切换。
   - 类型筛选：全部、普通 API、聚合供应商。
   - 排序：手动顺序、名称、延迟；默认手动顺序。
   - 供应商切换总开关。
   - 添加、第三方导入、刷新全部延迟。

2. 供应商卡片
   - 名称、当前状态、类型与协议。
   - 普通供应商展示 Base URL、模型数量和延迟。
   - 聚合供应商展示轮转策略、成员数和健康成员数。
   - 底部保留设为当前、测试/查看成员、编辑、删除。
   - 当前供应商使用蓝色强边框，健康状态继续使用绿色，延迟偏高使用琥珀色。

3. 排序行为
   - “手动顺序”下保留拖动排序并持久化现有顺序。
   - 名称或延迟排序只改变当前视图，不覆盖已保存的手动顺序。
   - 搜索和筛选同样不改写配置顺序。

4. 响应式
   - 宽屏三列卡片。
   - 中等宽度两列。
   - 应用允许的最小宽度下单列，工具栏自动换行。
   - 长名称、长 URL、英文界面和按钮文字都不能溢出或遮挡。

## 4. 实施步骤

### 阶段一：重构展示层

- 在不改后端命令和数据结构的前提下，将供应商工具栏、卡片和网格拆成职责清晰的展示组件。
- 在 `RelayScreen` 增加搜索、视图、筛选和临时排序状态。
- 复用现有 `Actions`、延迟检测和拖动排序逻辑。

### 阶段二：完成交互映射

- 将添加普通供应商、添加聚合供应商整合到添加菜单或并列动作中。
- 保留第三方导入菜单及刷新能力。
- 增加“刷新全部延迟”，并处理并发、加载态、部分失败和重复点击。
- 总开关关闭时禁止切换，但仍允许查看和编辑配置；行为与当前逻辑保持一致。

### 阶段三：视觉与无障碍

- 在 `styles.css` 中增加页面级样式，兼容亮色和暗色主题。
- 使用现有 `lucide-react` 图标；图标按钮提供 `title`/可访问名称。
- 明确当前、加载、失败、禁用、拖动和键盘焦点状态。
- 不改变全局品牌色变量，页面内仅用蓝色表达选择、绿色表达健康、琥珀色表达警告。

### 阶段四：验证与回归

- 静态检查：`npm run check`。
- 前端构建：`npm run vite:build`。
- Rust 回归：按仓库测试文档执行供应商配置、切换、轮转和导入相关 `cargo test`。
- Tauri 真实页面验证：亮色/暗色、最小宽度、1/3/12 个供应商、长名称和长 URL。
- 真实链路验证：切换普通供应商、切换聚合供应商、检查 `config.toml` / `auth.json`、延迟检测、测试请求、第三方导入、拖动排序和重启 Codex Deck。
- 确认“启动 Codex 时不自动改写配置”的现有承诺没有回归。

## 5. 验收标准

- 用户在一个页面内能找到、比较并切换供应商，无需先进入详情页。
- 当前供应商、供应商类型、协议、目标地址和健康状态可以快速扫读。
- 当前已有的添加、聚合、导入、排序、检测、测试、编辑、删除能力全部可达。
- 搜索、筛选和非手动排序不会意外改写供应商持久顺序。
- 总开关关闭时不会写入 Codex 实时配置文件。
- 亮色、暗色和应用最小宽度下不存在文字截断、控件重叠或横向失控。
- 自动化检查通过，并完成真实配置文件与页面业务链路验收后，才可声明测试通过。

## 6. 非目标

- 不改供应商数据模型、Tauri 命令或 Rust 配置生成逻辑。
- 不重做供应商详情编辑器。
- 不改全局侧栏和顶栏。
- 不加入 Cockpit Tools 的 Codex 账号、配额或多实例功能。
- 不复制参考项目源码或视觉资产。

## 7. 与总方案的关系

本页方案已并入全局总方案，以下内容作为供应商页面实现约束：

1. 供应商内容区使用实体管理模板，全局侧栏和顶栏使用总方案组件。
2. 保留手动拖动顺序；名称/延迟排序只影响当前视图。
3. 先迁移列表页，供应商详情随后使用总方案的详情页模板统一改造。

产品代码修改仍以用户明确回复“按总设计方案可以开发”为前提。

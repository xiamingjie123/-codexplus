# 设计：CodexPlusPlus 桌面端模型后缀提示

> 日期：2026-06-24  
> 状态：已批准，待实现  
> 依赖：阶段一 model catalog 原型后端已完成（`model_suffix` 模块 + `apply_model_catalog_to_config`）

---

## 1. 背景

阶段一已在后端实现 `model_list` 后缀语法（如 `deepseek-v4-pro[1M]`），应用 relay profile 时会自动生成 `model_catalog_json` 并写入 config.toml 指针，使 codex 客户端按模型识别真实上下文窗口。

但前端 `codex-plus-manager` 的“配置模型”和“模型列表”输入框目前没有任何提示说明支持后缀，用户不知道如何填写。本设计目标是以**最小改动**在前端暴露这一能力。

---

## 2. 目标

- 让用户知道可以在“配置模型”和“模型列表”中给模型名加后缀来指定上下文窗口。
- 保持现有输入框组件和交互不变。
- 前端不解析、不校验、不显示解析后的窗口值（决策权完全交给用户）。

---

## 3. 范围

**在范围内：**
- 更新 `apps/codex-plus-manager/src/App.tsx` 中两个模型相关输入框的 placeholder。
- 在两个输入框下方各新增一行 hint 文案。

**不在范围内：**
- 结构化 model_list 编辑器（阶段三）。
- 后缀实时高亮、校验、窗口可视化。
- 后端逻辑改动（已完成后端无需再改）。
- 新增依赖或组件。

### 4.3 文案约定

- 后缀示例统一使用 `[1M]`、`[200K]`、`[1000000]`。
- 明确说明“不写则使用 Codex 默认长度”，避免用户以为必须写后缀。

---

## 5. 数据流

前端仅做文案提示，不增加新的状态或 API 调用。

```
用户输入带后缀的模型名
        ↓
保存/应用 profile（现有流程）
        ↓
后端 parse_model_suffix / collect_catalog_entries / build_model_catalog_json
        ↓
generate catalog + 写 config.toml 指针（现有流程）
```

## 6. 测试计划

### 6.1 手工测试（使用 `npm run dev`）

桌面端手工测试使用开发模式，无需打包：

```bash
cd apps/codex-plus-manager
npm install
npm run dev
```

1. 在弹出的桌面窗口中打开一个 Relay Profile 编辑页。
2. 在“配置模型”输入 `deepseek-v4-pro[1M]`，保存。
3. 在“模型列表”输入：
   ```
   deepseek-v4-pro[1M]
   claude-sonnet-4[200K]
   qwen3-coder
   ```
   保存并应用 profile。
4. 检查 `~/.codex/config.toml` 包含 `model_catalog_json = "model-catalogs/<profile-id>.json"`。
5. 检查 `~/.codex/model-catalogs/<profile-id>.json` 中：
   - `deepseek-v4-pro` 的 `context_window` 为 `1000000`
   - `claude-sonnet-4` 的 `context_window` 为 `200000`
   - `qwen3-coder` 的 `context_window` 为 Codex 默认值或顶层 fallback
6. 运行 `codex debug models` 验证窗口正确。

### 6.2 大上下文窗口验证（以 100 万为例）

目标：验证 catalog 声明的大窗口值被 codex 接受，并在实际调用上游模型时不会触发上下文超限。

测试目标模型（通过 OpenRouter，配置自定义 provider）：
- `deepseek/deepseek-v4-flash`（100 万上下文）
- `qwen/qwen3-next-80b-a3b-instruct:free`（262K）
- `z-ai/glm-4.7-flash`（203K）
- `openai/gpt-oss-120b:free`（131K）

> 注意：OpenRouter API key 需自行配置，通过环境变量或 CodexPlusPlus 的 auth 内容注入，**不可硬编码到仓库**。

#### 方法 1：codex debug models 查看声明值

```bash
codex debug models \
  -c model="deepseek/deepseek-v4-flash" \
  -c model_catalog_json="/path/to/catalog.json"
```

确认输出中 `context_window` 为 `1000000`（或对应模型的目标值）。这是 catalog 生效的最轻量验证。

#### 方法 2：构造 10 万 token 级别输入验证压缩/发送

不需要真的构造 100 万 token。下载或生成一个约 10 万 token（几十 MB 英文文本）的公开测试数据集，例如：

- 维基百科长条目/书籍文本
- 公开的法律文档、技术手册
- Hugging Face `openai/openai-gpt-3` 等长文本评测数据

用 codex 对该文本做“总结前 10 万 token 的内容”，观察：
- 请求是否成功发出；
- `auto_compact` 是否按预期压缩；
- 上游是否返回上下文超限错误；
- 返回的 token usage 中 prompt token 数是否接近预期。

#### 方法 3：codex debug/verbose 日志查看实际 token 使用

在调用 codex 时开启 debug/verbose 模式（具体参数视 codex CLI 版本而定），检查日志中的实际 prompt token 数和模型看到的 `context_window`。

#### 方法 4：直接调用上游 API 验证上限

绕过 codex，直接用 curl/Python 调用 OpenRouter API：

```bash
curl https://openrouter.ai/api/v1/chat/completions \
  -H "Authorization: Bearer $OPENROUTER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek/deepseek-v4-flash",
    "messages": [{"role": "user", "content": "<10万token文本>"}]
  }'
```

确认上游对目标模型确实接受接近声明值的上下文。此方法验证的是上游能力，与 codex catalog 无关，用于排除“catalog 对了但上游不支持”的疑虑。

## 8. 风险与回退

| 风险 | 应对 |
|------|------|
| hint 文案在不同主题下对比度不足 | 使用项目现有 muted foreground 颜色变量 |
| placeholder 过长在某些窗口宽度下显示不全 | placeholder 保持简洁，完整说明放 hint |
| 用户误以为前端会校验后缀 | hint 文案不承诺校验，保持“支持写”而非“必须写对” |

### 大上下文窗口验证

#### 方法 1：codex debug models 查看声明值 ✅

使用 example 生成含后缀的 catalog：

```bash
cargo run -p codex-plus-core --example generate_model_catalog -- \
  "DeepSeek-V4-Flash[1M]" \
  "Kimi-K2.6[262K]" \
  "MiniMax-M2.7[205K]"
```

`codex debug models` 输出：

```
DeepSeek-V4-Flash 1000000 100
Kimi-K2.6 262000 100
MiniMax-M2.7 205000 100
```

结论：catalog 声明值被 codex 正确接受，每个模型有独立的上下文窗口。

#### 方法 4：直接调用上游 API 验证上限 ✅

端点：`https://ark.cn-beijing.volces.com/api/coding/v3`

三个模型直接调用均成功：

| 模型 | 返回 model | 状态 |
|------|-----------|------|
| DeepSeek-V4-Flash | deepseek-v4-flash | ✅ 200 |
| Kimi-K2.6 | kimi-k2.6 | ✅ 200 |
| MiniMax-M2.7 | minimax-m2.7 | ✅ 200 |

#### 方法 2：构造大输入验证压缩/发送 ✅

构造约 160k token 的重复文本输入，分别调用三个模型：

| 模型 | prompt_tokens | 结果 |
|------|--------------|------|
| DeepSeek-V4-Flash | 160012 | ✅ 成功 |
| Kimi-K2.6 | 160018 | ✅ 成功 |
| MiniMax-M2.7 | 160049 | ✅ 成功 |

结论：三个模型均能处理约 160k token 的输入，未返回上下文超限错误。由于网络/代理层对超大 payload 的限制，未继续压测到 1M；但 160k 已覆盖 Kimi（262K）和 MiniMax（205K）标称值的大部分，DeepSeek（1M）的超大窗口需在有更好网络环境时进一步验证。
